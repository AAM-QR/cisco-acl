Forgetting the implicit deny
Every ACL ends with an invisible deny any. If you only add permit rules and forget to consider what gets blocked, everything else is silently dropped.
access-list 10 permit 192.168.1.0 0.0.0.255
! Everything else is denied — even your own traffic!
Always add a permit ip any any at the end if you want everything else through.
Wrong wildcard mask direction
Wildcard masks are the inverse of subnet masks. A /24 network uses wildcard 0.0.0.255 — not 255.255.255.0. Using the subnet mask instead of the wildcard is one of the most common errors.
! WRONG: access-list 10 permit 192.168.1.0 255.255.255.0
! RIGHT: access-list 10 permit 192.168.1.0 0.0.0.255
Subnet mask + Wildcard mask = 255.255.255.255. Use the Wildcard Calc tab to check.
Applying ACL in wrong direction
Inbound ACLs filter traffic before the router processes it. Outbound filters after. Applying an inbound ACL when you meant outbound (or vice versa) blocks the wrong traffic.
! Inbound on Gi0/0 — filters before routing
ip access-group 100 in
! Outbound on Gi0/1 — filters after routing
ip access-group 100 out
Standard ACLs should be placed closest to the destination. Extended ACLs closest to the source.
Rule order / shadowing
ACLs are processed top-down and stop at the first match. A broad permit/deny early in the list can shadow (completely block) more specific rules below it.
10 permit ip any any ← matches everything!
20 deny tcp any any eq 23 ← NEVER reached
Put specific rules before general ones. This simulator highlights shadowed rules in grey.
Standard ACL placed too far from destination
Standard ACLs only filter on source IP. If placed near the source, they can block traffic to ALL destinations, not just the one you intended.
! Trying to block access to server 10.0.0.5
! Standard ACL on Gi0/0 (ingress, near source)
! This blocks 192.168.1.0/24 from going ANYWHERE
Place standard ACLs on the interface closest to the destination to limit the blast radius.
